Final Blog (12/1/2020)

For this week, I fixed several bugs with the tool and demo'd my project with a stakeholder. In addition, I ran into a roadblock with the Unity Engine that prevented me from implementing an export feature.

Export (3 hours)

One of the features are wanted for the tool was the ability to export user meshes to an external file format. I tried 2 formats: obj and fbx. Obj format does not work well since it stores no information regarding a meshes materials to the file; only physical information such as the list of verticies to construct the mesh. Given this, FBX format is ideal. Unity has an FBX export package that works nicely. However, I ran into two additional issues:

  1. Whenever I attempted to export the mesh during play mode in the Unity editor, the plane generated would be missing its materials. I believe this issue is due to file serialization: the FBX exporter may be using the underlying files on the game objects to generate the mesh. This means that any modifcations during play mode (such as the material information) may not be written to disk. The exporter works fine outside of runtime however.
  2. When I attempted to use another FBX exporter tool, which supports runtime mesh conversion, webGL ran into an issue discribed below:
The tool is having an issue finding a path to write the .fbx file to. After looking up the issue, I discovered that WebGL has no means to interface with the user's local file system. This means that saving the mesh to any file system is not possible with WebGL alone. To resolve this, I would have to either write my own file system interface, or look into writing javascript code on the server side to enable this functionality. However, with my final presentaiton coming up in a few days, this is not feasible. The export feature will have to be tabled for now.
